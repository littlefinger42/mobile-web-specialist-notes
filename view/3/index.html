Introduction

Its important to make logical tab order so users can navigate the website without use of a keyboard. You may want to specify a tab order that follows relationships in the content, rather than following the order in the code.

Examples

W3 Accessibility Tab Order Webaim

How?

An alternative order can be specified using the tabindex attribute of the interactive element. The tabindex is given a value between 0 and 32767. Elements that have a tabindex value higher than zero will receive focus before elements without a tabindex or a tabindex of 0.

A tabindex usually has three distinct uses:

  • tabindex="1" - (or any number > 1) Defines explicit tab order. Generally a bad idea.
  • tabindex="1" - (or any number > 1) Allows any element to recieve keyboard focus if it has this attribute. Places element in the logical navigation flow.
  • tabindex="1" - (or any number > 1) Removes from tab order and allows elements to recieve a programmatic focus, set by JS.

Introduction

The main content or navigation isn't usually the first thing on the page. Keyboard and screen readers must navigate through a lot of elements before they get to where they want to be. A skiplink is the first element on the page, and is usually hidden. The screenreader or keyboard user can focus on this element and use it to navigate to the content or the navigation bar, thus saving time.

How?

Create a anchor tag element as the first element, straight after the <body> opening tag. Then make it visibility:hidden, unless focus is on it. 

<body>
    <a class="skiplink" href="#maincontent">Skip to main content</a>
    ...
    <main id="maincontent">
    <h1>Heading</h1>
    ...


.skiplink {
    visibility: hidden;
}
.skiplink:focus {
    visibility: visible;
}

Introduction

Accessible Rich Internet Applications (ARIA) are additional attributes to add to elements to define roles, values and traits. These attributes are used to make the web content more accessible. It is mostly used to define a role.

Roles

Roles describe widgets that aren't otherwise available in HTML4. Some roles have their own elements in HTML5. Where available, HTML5 semenatics should be preferred over HTML4 roles. There are more than 30 ARIA roles, including: alert, checkbox, dialog, menubar, slider, tooltip. Roles don't actually do anything themselves, they just provide the screen reader with more information.

<div class="btn btn-default" role="button">Submit</div>

States

States describe the current interaction state of an element. There are 9 states. For example: aria-checked, aria-disabled, aria-invalid

<div aria-expanded="true">Menu</div>
Will output "Menu Expanded" by a screenreader.

Properties

Properties describe the characteristics of these widgets. There are more than 20 ARIA properties including: aria-describedby, aria-label, aria-required, aria-draggable

<div aria-label="say this">Say that</div>
Will output "say this" by a screenreader.

Live Regions

Dynamic content which updates a page without reload, and isn't interactive, is a live region. Live regions have their own ARIA properties:

  • aria-live - Used to set the priority with which screenreader should treat updates to this region. Possible settings are:
    • Off - (default) User wont recieve any notification of an update.
    • Polite - Will wait until the user pauses, before announcing an update.
    • Assertive - Wont wait. Will announce the update instantly.
  • aria-controls - Used to associate a control with the regions it controls. Regions are identified like an ID in a div. Multiple regions can be controlled using a space, eg aria-controls="myRegion1 myRegion2"

An example can be found on MDN

Level A WCAG Requirements

Colour should not be used as the sole method of conveying content or distinguishing visual elements. (links should not be identifiable just by the colour). For links, if you don’t provide any visual que that isn’t colour for links, the colour contrast between text should be at least 3:1. For links, they must have a non-colour designator for focus/hover states.

Level AA WCAG Requirements

The foreground and the background colour should have a 4.5:1 contrast ratio.

Therefore, to be AA compliant, the page must have the following:

  • 4.5:1 Contrast between non-link text colour and the background
  • 4.5:1 Contrast between link text colour and the background
  • 3:1 Contrast between link text colour and surrounding text colour
This becomes more complicated if you want to provide various link states (hover/focus, visited, active) colous. W3 has a list of colours which fit these criteria for links.

Names & Roles

Every element which can be interacted, should have a name and a role. If the user is expected to put an input, it should also have a value.

Name

  • Form field - The name for a form field is it's label. <label for="text_1">Fruit type</label><input id="text_1" type="text" value=""/>
  • Link - The name for an anchor tag is it's text, or alt tag. <a href="/home.html">Home</a>
  • Button - The name for a button is the text, or the value. <button>Home<button>

Role

Roles are determined by the HTML element. It determines the standard actions and properties. You can also use the type="" attribute on <input> to set the role of an element. Eg <input type="button" value="save">
